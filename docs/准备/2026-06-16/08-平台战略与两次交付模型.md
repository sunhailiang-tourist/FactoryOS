## 一、战略定位（当前阶段）

### 1.1 我们在做什么

| 维度 | 定义 |
|------|------|
| **产品** | 面向制造业的 **AI 执行平台地基**（FactoryOS），不是聊天 Agent，不是 ERP 替代品 |
| **首要目标** | 技术基座 **成熟、可靠、值得信任**，直击制造业通病 |
| **验证方式** | 哈森体系内灯塔部署（自有工厂 + 旗下子公司），积累实战经验后再见老总授权推广 |
| **当前不做** | 对外营销、噱头叙事、行业垂直 SKU 包装（属飞轮产出物，非地基定义） |

### 1.2 核心命题

```text
平台地基（全行业通用机制）
    → 第一次交付（通病包，一次结案）
    → 第二次交付（该厂特异痛点，平台延伸）
    → 沉淀 Pack → 下一家「第一次」更快
    → 数据飞轮
```

**见老总的一句话**：

> 我们不是来换 ERP，也不是来卖 AI 概念；第一次用同一套平台机制打掉制造业都会遇到的执行断层；第二次在同一地基上长你们厂特有的能力；两次之后，第三家会更快——这就是技术复利。

### 1.3 行业印证（策略审计结论）

本策略与 Celonis（Overlay + land-and-expand）、Tulip（Composable overlay）、黑湖（轻量采集 → 组件化 → 多厂复制）、工业 SaaS Land-and-Expand  playbook **同构**。详见 [07-行业痛点与顶级平台差距分析](./07-行业痛点与顶级平台差距分析.md) §九。

**近最优解条件**（缺一不可）：

1. 通病包边界极窄、可验收  
2. 第一次交付有书面结案（≤90 天）  
3. 第二次定制 ≥70% 入库为 Pack  
4. 内部灯塔厂按外部客户标准验收  
5. Shadow + 对账 + Revert 先于规模写操作  

### 1.4 部署即资产化（顶级战略 · 大白话）

> **定版口径（2026-06-16）**：与 §1.2 飞轮命题同级，对内对外可原句复述。

**大白话**：

> **每部署一家厂，就是在现场飞速提炼老系统里的核心业务，变成我们的 Graph、Pack 和 Implementation Package——这就是我们的数字资产；第一家像考古，第二家起像 import，所以越部署越快。**

**正式表述**：

| 维度 | 定义 |
|------|------|
| **我们在提炼什么** | 老系统中的 **业务执行链路**（谁报工、写哪、谁确认、怎么对账、怎么撤），不是 ERP 全量数据副本 |
| **怎么提炼** | Connector 读接口/结构化摸底（L2）→ 业务发现出 Graph Draft（L3）→ 工作坊人工 freeze（L4）→ D1/D2 结案 **export Package** |
| **变成什么资产** | frozen Graph · Rule 模板 · Skill Pack · Connector Pack · Implementation Package |
| **为什么越来越快** | 第 2 家起 **import** 为主、workshop 为差量；相似 D2 直接选用 Pack，不必重复集成 |
| **红线** | **≠ 扒库做数仓**（Data-L3 选配）；**≠ AI 自动认定链路**（必须 freeze）；**≠ 改内核**（扩展只增 Pack/Graph 新版本） |

```text
老系统（ERP/MES/API/单据）
    → 部署过程提炼（读 + 工作坊 + Shadow 验证）
    → frozen Graph + Pack
    → export Implementation Package
    → 下一家 import → 周期缩短（飞轮）
```

**与交付模型关系**：本战略是 **D1/D2 + R4 export** 的顶层归纳；工程步骤见 [04](./04-工厂实施手册.md) Step 1～2；数据边界见 [06](./06-数据边界与部署策略.md)（L2 采集 ≠ 全量入库）。

## 二、平台地基：制造业通病机制（与行业无关）

地基不是「某行业 MES 功能合集」，而是**任何制造厂都会撞到的结构问题**及其**固定解法**。

### 2.1 通病 ↔ 平台机制对照

| 制造业通病 | 平台自带机制 | 技术模块 |
|------------|--------------|----------|
| 计划与执行脱节（Execution Gap） | Overlay + Graph 跨系统编排 | graph_service, agent_orchestrator |
| ERP/MES/WMS/OA 孤岛 | Connector + MCP 稳定动词 | connector_sdk, MCP Registry |
| 现场不愿用系统、采集难 | Harness Loop + 轻入口（IM H5/语音/扫码） | agent_orchestrator, h5-worker |
| AI/自动化不敢写库 | DSL + Rule Engine，Agent 无写权限 | rule_engine, execution_service |
| 错了收不回来 | Audit + Revert + Compensator | audit_service, execution_service |
| 现场数与账本对不上 | 对账 Job + Shadow Mode | reconciliation, dry_run |
| 每上新厂从零集成 | Implementation Package export/import | graph/rule/skill packs |

### 2.2 技术分层（Coding 不变量）

```text
Layer 0  信任内核（一次建设，全行业复用）
         Graph + Rule + DSL + Audit + Revert + ExecutionRecord

Layer 1  MCP Gateway
         稳定动词 + Registry → Connector

Layer 2  Skills
         场景剧本（通病 Skill / 定制 Skill）→ 绑定 Graph Pack

Layer 3  Harness Loop
         感知 → 授权 → 沙箱 → 确认 → 执行 → 验证 → 记录

Layer 4  Evolution（后期）
         仅优化 Skill 提示/步骤；不改 frozen Graph/Rule
```

**原则**：换 ERP、换 MES、换行业，**Layer 0–3 架构不变**；变的是 Connector 与上层 Pack。

### 2.3 与工程 Phase 的关系

| 工程 Phase（见 03） | 战略角色 |
|---------------------|----------|
| **Phase 0** | 冻结地基契约（Schema / ADR / Graph v1） |
| **Phase 1** | 建成通病包 v1 可运行内核 + 第一家第一次交付的技术前提 |
| **Phase 2** | 读分析 + 主管 Skill；为第二次交付常见扩展铺路 |
| **Phase 3** | License、更多 DSL、Graph Builder 辅助 |
| **Phase 4** | 多厂复制、Integration Studio、Pack 市场 |

> **注意**：工程 Phase ≠ 客户交付次数。Phase 1 代码完成后，才具备对灯塔客户做「第一次交付」的条件。

## 三、两次交付模型

### 3.1 总览

```text
┌────────────────────────────────────────────────────────────┐
│                    平台地基（Layer 0–3）                     │
└────────────────────────────┬───────────────────────────────┘
                             │
           ┌─────────────────┴─────────────────┐
           ▼                                   ▼
    ┌──────────────┐                    ┌──────────────┐
    │  第一次交付   │                    │  第二次交付   │
    │  通病包 D1   │                    │  扩展包 D2   │
    ├──────────────┤                    ├──────────────┤
    │ 固定范围     │                    │ 该厂特异规则  │
    │ ≤90 天结案  │                    │ Graph/Skill  │
    │ Shadow→上线 │                    │ 扩展         │
    │ 书面验收     │                    │ ≥70%→Pack    │
    └──────┬───────┘                    └──────┬───────┘
           │                                   │
           └─────────────┬─────────────────────┘
                         ▼
                  Implementation Package
                  → 下一家 D1 加速
                  → 相似 D2 降级为 Pack
```

### 3.2 第一次交付（通病包 D1）

**目标**：任何制造厂，**不替换 ERP/MES**，用同一套机制打掉共性执行断层。

**通病包 v1 范围（锁定五项，不多不少）**

| # | 能力 | 验收要点 |
|---|------|----------|
| D1-1 | **跨系统只读查询** | 工单/报工历史可查（`QUERY_WO`, `QUERY_REPORT_HISTORY`） |
| D1-2 | **Governed 写入** | 仅经 DSL + Rule 的写操作（如 `WORK_REPORT`） |
| D1-3 | **全链路审计** | 每次写操作有 ExecutionRecord，append-only |
| D1-4 | **可回滚** | `WORK_REPORT_REVERT`；24h 内可演示恢复一致 |
| D1-5 | **每日对账** | MES/ERP 与 FactoryOS 执行记录零 drift 或自动告警 |

**标配 Skill**：`work_report_v1`（工人报工 + 主管确认）——作为 D1 的**第一个具体 workflow**，不是地基的全部。

**第一次交付标准步骤**（对齐 [04-工厂实施手册](./04-工厂实施手册.md)）

```text
Step 0  准入评估（1–3 天）
Step 1  环境接入（只读）
Step 2  Graph 冻结（通病链路，5–8 节点）
Step 3  Rule + DSL 绑定
Step 4  Shadow Mode（≥2 周，写操作零生产）
Step 5  单线/小范围 UAT
Step 6  灰度 + 培训
Step 7  运维移交 + D1 验收签字
```

**第一次交付 Gate（全部满足才结案）**

- [ ] Shadow ≥14 天，产出《影子运行报告》
- [ ] D1-1～D1-5 五项验收通过
- [ ] 实施方 + 工厂负责人 **D1 签字**（模板见 04 §七）
- [ ] **export** Implementation Package v1（Graph/Rule/Skill/Connector 配置/对账样例）
- [ ] **未**将工厂特异规则写入平台内核

**周期**：首家（含 Phase 0–1 研发）6–9 月；同模板第二家 D1 **4–8 周**。

### 3.3 第二次交付（扩展包 D2）

**目标**：吃掉该厂剩余约 **50% 长尾痛点**（计件规则、质量追溯、外协、设备信号、特殊审批等），**不推翻 D1**。

**边界**

| 允许 | 禁止 |
|------|------|
| 新增 Graph 节点/边（新版本 freeze） | 修改已 frozen 的 D1 Graph 语义 |
| 新增 Skill / DSL 动作（经 ADR） | Agent 直写 ERP/MES |
| Connector 字段扩展 | 无 Audit/Revert 的写操作 |
| 客户私有 Rule 分支 | 100% 一次性 bespoke 代码不入库 |

**D2 典型扩展项（按优先级，非一次全做）**

| 优先级 | 扩展项 | 沉淀为 |
|--------|--------|--------|
| P1 | 计件/工资草案 Skill | `skill-piece-rate-draft` |
| P1 | 工人端进度回灌（本单/本工序状态） | `skill-worker-status` |
| P2 | 主管班次指挥（批量确认/驳回） | `skill-supervisor-shift` |
| P2 | 轻量质量追溯（批次/人/工序绑定） | `graph-quality-trace-v1` |
| P3 | Excel/微信只读 Connector | `conn-excel-wechat-read` |
| P3 | 异常 Andon + IM 通知 | `skill-andon-notify` |

**第二次交付 Gate**

- [ ] D2 范围事前书面确认（与 D1 无重叠争议）
- [ ] 新 Graph 版本 freeze + 签字
- [ ] D1 能力回归测试通过（无退化）
- [ ] **≥70%** D2 实现进入共享 Pack；≤30% 客户私有配置
- [ ] export Implementation Package v2
- [ ] 《D2 验收报告》+ 签字

**周期**：在 D1 结案后 **4–10 周**（视扩展复杂度）。

### 3.4 第三次及以后：飞轮生效

| 场景 | 行为 |
|------|------|
| 第 3 家厂，痛点与第 1 家 D2 相似 | D2 能力直接从 Pack 选用，计入「第一次」或轻量「第二次」 |
| 新特异场景 | 仍走 D2 扩展，但周期缩短（地基与 Pack 库已厚） |
| Pack 累计足够 | 对外可选用行业 Solution Pack 命名（**飞轮产出**，非现在定义地基） |

## 四、五条战略铁律（防失败）

| # | 铁律 | 防什么 | 行业依据 |
|---|------|--------|----------|
| **R1** | 通病包 v1 **只有五项**（§3.2） | 第一次交付变 SI 项目 | Tulip/Celonis 单点高影响；Unusual VC 灯塔绑架 roadmap |
| **R2** | 第一次交付 **≤90 天书面结案** | pilot purgatory | McKinsey/WEF；Springboard35 结构化 pilot |
| **R3** | **第二次**才开放工厂特异 Graph/Skill | 内核被定制腐蚀 | 黑湖组件化；Celonis land-and-expand |
| **R4** | 每家厂结束必须 **export Implementation Package** | 飞轮转不起来 | Tulip 模板；Deploy95 部署知识复利 |
| **R5** | 内部子公司 **按外部客户验收**，不无限加需求 | ADAMOS 式永远试点 | Carl Vause：无预算/时间线 = 探索而非交付 |

## 五、灯塔部署：哈森体系内验证路径

### 5.1 角色定义

| 角色 | 用途 |
|------|------|
| **研发侧（哈森制造/精密 + 平台团队）** | Phase 0–1 建设地基；产出可部署制品 |
| **灯塔厂 A** | 第一次 D1 全流程；产出 Playbook v1 |
| **灯塔厂 B** | 验证 D1 复制速度；可选承担 D2 扩展试验 |
| **子公司老总** | 授权、资源、UAT 签字；不是「无限试点」赞助人 |

### 5.2 推荐节奏

```text
T0        Phase 0 锁定（2–3 周）
T1–T2     Phase 1 研发（6–8 周）→ 通病包可运行
T3        灯塔厂 A · D1（≤90 天，含 Shadow）
T4        灯塔厂 A · D2（4–10 周，选 1–2 项扩展）
T5        灯塔厂 B · D1（目标 4–8 周，import Package）
T6        复盘 → 集团推广决策 / 对外复制准备
```

### 5.3 见老总前的「及格线」（无需营销话术）

1. 接真实 **Path A：ERP（写+读）+ 钉钉**（或 Path B：MES+ERP），非纯 mock  
2. 《影子运行报告》≥2 周  
3. 现场演示一次 **Revert**，数量拉回一致  
4. D1 五项 Gate 勾选表  
5. 第二家复制清单（Implementation Package 可 import）

## 六、数据飞轮机制

> **飞轮本质**（见 §1.4）：每次部署都在把 Legacy 核心业务提炼为 Pack；§6.1～6.3 是运行期与入库的细化。

### 6.1 飞轮输入（Layer 1 数据，见 06）

- ExecutionRecord / Audit 日志  
- 对账 Job 结果（drift 模式）  
- Revert 记录与原因  
- Shadow 与生产的偏差统计  
- D2 定制项及最终 Pack 化比例  

### 6.2 飞轮输出

| 输入模式 | 沉淀物 |
|----------|--------|
| **部署提炼（§1.4）** | Graph / Connector / Skill **Pack**；Implementation Package |
| 高频 revert 原因 | Rule 模板补丁 |
| 重复出现的 Graph 分支 | Graph Pack 草案 |
| 相似 D2 扩展 | 新 Skill Pack |
| 新 ERP/MES 对接 | Connector Pack |
| 实施摩擦 | 04 手册 / Console 功能 backlog |

### 6.3 Pack 入库标准

一项 D2 扩展 **只有** 在满足以下条件时才进入共享 Pack：

1. 已在 ≥1 家厂生产环境运行 ≥30 天  
2. 有文档 + 验收用例 ≥5 条  
3. 不破坏 Layer 0 红线（ADR-002）  
4. 可配置启用，非硬编码客户 ID

## 十一、一句话战略
