## 一、ChatGPT 原始对话：核心问题与结论

### 1. 产品定位

| 问题 | 结论 |
|------|------|
| 做 Agent 还是做 ERP？ | 做 **企业业务操作系统（FactoryOS）**；Agent 只是交互入口，不是产品本体 |
| 与金蝶/用友等关系？ | **AI Overlay**：不替换、不改造 ERP/MES/WMS/OA，只接管人机交互与跨系统编排 |
| 核心价值是什么？ | **Business Graph（业务图谱）** + 规则引擎 + 可控执行；多模态（语音/拍照）是入口，不是壁垒 |
| 品牌命名？ | 对外主品牌 **FactoryOS**；`Factory-AI-Native-System` 仅作内部架构名 |

### 2. 客户与业务类型

| 问题 | 结论 |
|------|------|
| 客户怎么分？ | **A 类**（有 ERP/MES，优先）、**B 类**（烂系统+Excel，痛点强但数据难）、**C 类**（无系统，短期不做） |
| 产品线怎么分？ | **产品 A：Agent Overlay**（难度约 100）；**产品 B：AI Factory OS 全栈**（难度约 1000，暂缓） |
| 制造业真正痛点？ | 系统多、数据散、培训贵；人在系统间**搬信息**，不是系统缺功能 |
| 免费策略？ | Agent 可免费引流；**部署、定制、私有化、流程改造** 收费 |

### 3. 场景与落地顺序

| 问题 | 结论 |
|------|------|
| 先做哪些场景？ | **工人报工** → **生产主管分析** → **老板决策**（均先只读/建议/确认） |
| 首期禁止做什么？ | 自动财务记账、付款、成本结转、库存调整、无确认自动审批 |
| 操作原则？ | `先读 → 后建议 → 再确认 → 最后自动化` |
| 哈森第一年目标？ | 1 条真实产线跑通 **Graph + Agent + ERP/MES Overlay** 闭环 |

### 4. 工程设计（对话审定版）

| 问题 | 结论 |
|------|------|
| 优先级排序？ | `可靠性 > 安全性 > 可追溯 > 可回滚 > AI 能力` |
| 核心架构？ | 多模态 UI → Agent Orchestrator → **Business Graph** → **Rule Engine** → **Execution DSL** → Connectors |
| 完整工程分层？ | **12 层**（接入→采集→发现→人工确认→孪生→规则→DSL→编排→审计→沙箱→回滚→执行） |
| Agent 能否直写库？ | **禁止**；必须 `Agent → DSL → Rule Engine → API → ERP` |
| Graph 能否 AI 全自动？ | **不能**；AI 出草稿，业务负责人确认后 **freeze** |
| Graph 要不要版本？ | 要；`V1.0 / V1.1 / V2.0`，冻结后不可原地改 |
| Agent 安全分级？ | L0 只读 → L1 建议 → L2 执行+确认 → L3 低风险自动 |
| 核心投入顺序？ | Graph → Rule Engine → Audit → Rollback → Agent 编排 → 多模态（与常见团队相反） |
| 最大风险？ | 不是模型不够聪明，而是数据错、链路错、权限错、执行错、**回滚失败** |
| 战略评分（对话）？ | 技术 8、客户价值 9、商业 9、可落地 7.5、战略 9；风险中等可控 |

### 5. 阶段演进（对话）

| 阶段 | 目标 |
|------|------|
| 2026–2027 | 制造业 Agent Overlay，接 ERP/MES/WMS/OA |
| 2027–2028 | Business Graph Builder（半自动） |
| 2028–2030 | Agent Workflow Platform（客户自配置） |
| 2030+ | Legacy 退化为数据存储层，FactoryOS 成入口 |

### 7. 业务拆分与工程设计是否合理？

| 问题 | 结论 |
|------|------|
| 业务类型拆分是否合理？ | **基本合理**；A/B/C 分型、产品 A/B、场景优先级逻辑成立 |
| 工程设计是否合理？ | **原则 9/10**；6 层/12 层方向正确，MVP 需裁剪，勿按 12 层齐步走 |
| 商业潜力是否可靠？ | **赛道真实、痛点真实**；对话 9 分略乐观，实际约 **7–8 分**（前期偏项目制/集成收入） |
| 可落地性是否可靠？ | **分阶段可靠**；第一年报工闭环可信；「3 天 Graph 上线」是愿景非交付承诺 |
| 需修正的认知？ | A 类「低难度」应改为**中等**；B 类数据比 A 更难；第一个客户 **6–9 个月**更现实 |

### 8. 业务链路锁定 + 可回滚（工程核心）

| 问题 | 结论 |
|------|------|
| 链路锁定是什么？ | 保证 Agent **按正确、已审批的流程**执行 |
| 怎么锁定？ | **三层**：frozen Graph 版本 + 绑定 RuleSet + 封闭 DSL/Agent Module；执行链门禁 |
| Graph 怎么 freeze？ | draft → in_review → **frozen**；带 checksum；未 freeze 拒绝执行 |
| Agent 边界？ | 只产出 DSL 计划；**无 ERP 写权限**；固定 FSM，非 open-ended |
| 回滚是什么？ | 对**单次写操作**补偿/撤销，不是 undo 整个 Graph |
| 怎么回滚？ | 每个写 DSL 预注册 **Compensator**；`ExecutionRecord` + before/after 快照；Saga 逆序补偿 |
| MVP 最小集？ | 4 个 DSL：`QUERY_WO`、`WORK_REPORT`、`WORK_REPORT_REVERT`、`QUERY_REPORT_HISTORY` |
| 一句话？ | **锁定靠 frozen Graph + Rule + DSL；回滚靠 ExecutionRecord + Compensator，Undo 走同一 Executor** |

### 9. 全程 AI Coding 的锁定策略

| 问题 | 结论 |
|------|------|
| 总体形态？ | **Modular Monolith**（先单体按模块分包，微服务 Phase 3 后再拆） |
| 北极星场景？ | 哈森 1 产线 × **工人报工 Agent** × **Path A ERP 写+钉钉**（平台仍支持 Path B MES） |
| 技术建议？ | TS 或 Python 后端 + OpenAPI 先行 + PostgreSQL；spec → test → code |
| Coding 前先锁什么？ | 5 份 JSON Schema + ADR + 验收用例 |
| Phase 0（2–3 周）？ | 规格锁定：ADR、Graph v1.0 freeze、Schema、OpenAPI、Connector 调研 |
| Phase 1（6–8 周）？ | 报工闭环：audit → execution → graph → rule → agent → **真实 ERP write**（灯塔）→ revert |
| 详见 | [03-锁定实施策略](./03-锁定实施策略.md) |

### 10. 其他工厂快速接入 ERP/OA 的实施过程

| 问题 | 结论 |
|------|------|
| 「快速」多快？ | 第 1 家 6–9 月；第 2 家同 ERP **4–8 周**；新 ERP 厂商 **+4–8 周** |
| 标准七步？ | 准入评估 → 只读摸底 → **Graph/Rule freeze** → 影子对账 → UAT → 灰度 → 运维 |
| 准入分级？ | S（快）/ A / B / C（不接）；只卖**已认证 Connector 组合** |
| OA 首期？ | 多为**通知 + 只读审批**；自动流转 Phase 3 |
| 快速复制靠什么？ | Connector 认证列表 + 行业 Graph/Rule 模板 + **Integration Studio** |

### 11. 模块化、商业化、各端入口

| 问题 | 结论 |
|------|------|
| 业务链路能否模块化？ | **能且应该**；三层：**Graph Pack / Capability Pack / Connector Pack** |
| 免费 vs 收费？ | 免费：L0 只读 + 基础入口；收费：写操作、Revert、Connector、多厂、私有化 |
| 各端用户入口？ | 工人/主管 → **钉钉或企微 H5**；财务/IT/实施 → **PC Web**；可选工位屏 |
| Phase 1 是否做 App？ | **不建议**；除非客户无钉钉/企微 |
| 最佳组合？ | **「1+2+1」**：IM 微应用（主）+ PC 管理台（必）+ 可选工位屏/小程序 |
| 详见 | [05-模块化与商业化方案](./05-模块化与商业化方案.md) |

### 12. 业务数据跟踪 vs 只做部署

| 问题 | 结论 |
|------|------|
| 要不要做客户全量业务数据跟踪？ | **默认不做**；不做平台级业务数仓、不跨客户汇总 |
| 只做部署够不够？ | **不够**；必须做部署 + **运行跟踪**（Layer 1）+ 对账 |
| 推荐模式？ | **部署 + 运行平台**（非纯实施商、非默认数据中台） |
| 业务数据权威在哪？ | **Layer 0**：ERP/MES 为账本；FactoryOS 不替代 |
| FactoryOS 必存什么？ | **Layer 1**：Graph/Rule、Audit、ExecutionRecord、对账、实施签字 |
| 可选缓存？ | **Layer 2**：最小字段、TTL；能读 ERP 就不落库 |
| 何时做数据池？ | **Layer 3**：仅 C 类/产品 B 或客户选配分析模块 |
| ChatGPT「数据跟踪/数据池」？ | 跟踪 = A+B 执行与查询；数据池 = C 类/产品 B，非 Overlay 默认 |

## 三、已达成共识的「铁律」汇总

1. 卖 OS，不卖聊天机器人；Agent = UI 层。  
2. 先做 Overlay，不做 ERP 替代；C 类客户、产品 B 暂缓。  
3. AI 不认定最终业务链路；Graph 必须人工确认并 freeze。  
4. 写操作必经 DSL + Rule Engine；Agent 禁止直写 ERP/MES。  
5. 先读再写；首期场景：报工 → 分析 → 决策。  
6. 每个写 DSL 要有 Compensator + ExecutionRecord + 快照。  
7. MVP 裁剪 12 层，不要一次做全平台。  
8. 实施 = Graph freeze 签字 + 对账 + UAT，不是「连上 API 就上线」。  
9. 商业化按模块授权（Graph / Capability / Connector），不是卖一个 App。  
10. 业务事实在 ERP/MES；FactoryOS 默认不做全量业务数据托管，必做执行跟踪与对账（见 06）。  
11. 文档修改需负责人确认后再动。  
12. 地基解决制造业通病；**第一次交付通病包、第二次交付扩展包**；定制 ≥70% 沉淀 Pack（见 08）。  
13. 内部灯塔厂按外部客户验收，第一次 ≤90 天书面结案（见 08 R2/R5）。  
14. **部署即资产化**：部署过程提炼老系统核心业务为 Graph/Pack，第一家考古、第二家起 import 加速（见 08 §1.4）。
