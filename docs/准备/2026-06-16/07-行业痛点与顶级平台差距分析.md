| 痛点 | 表现 | 业内依据 |

## 三、FactoryOS 方案覆盖度对照

| 痛点簇 | 现有设计（00–06 + MCP/Skills/Harness） | 覆盖度 |
|--------|----------------------------------------|--------|
| ERP/MES 孤岛、人搬数据 | Overlay + Connector + Graph | **高** |
| Execution Gap | 跨系统 Agent + 对账 | **高** |
| 工人不会用 MES | 语音/拍照 + 钉钉 H5 + Skills | **高** |
| 写操作不可靠 | DSL + Rule + Revert + Audit | **很高**（核心差异化） |
| 试点地狱 / 不可复制 | 七步实施 + 模板（**产品未建**） | **中** |
| 系统是记录非赋能 | 文档偏采集；**现场信息回灌弱** | **中偏低** |
| 计件工资/劳资争议 | 报工可触及；**无计件薪酬引擎** | **低–中** |
| 质量追溯 | 文档提及；**无专项模块** | **低** |
| 外协/排产 APS | 明确暂缓 | **刻意不做**（合理） |
| AI 数据差 | 人工 freeze Graph；非 process mining | **中** |
| ROI 证明 | 未产品化 | **低** |
| 与金蝶/用友竞争 | 中立 Overlay 叙事有；**产品证据无** | **中** |
| MCP/Skills/Harness | **已入 Phase 1**（ADR-004、03、14）；Harness P1 后半 | **设计有、工程无** |

**小结**：在「跨系统 + 现场易用 + 可控执行」上对齐行业最难痛点；在「鞋服垂直闭环（薪+质+外协）」与「商业证明」上仍有明显空白。

## 四、冲「商业 / 业务 / 可落地」顶级的差距清单

### 4.1 商业价值要顶级 — 还差什么

| # | 缺口 | 为何关键 | 建议补齐 |
|---|------|----------|----------|
| B1 | **可销售的楔子 SKU** | 「FactoryOS」过大，客户不知买什么 | 对外单品：**鞋服「报工+计件协同」**；对内仍用 FactoryOS 内核 |
| B2 | **量化 ROI 仪表盘** | 65% 数据问题 + ROI 难证阻碍采购 | 内置指标：报工耗时、对账一致率、核算周期、差错率 |
| B3 | **标杆案例一页纸** | 顶级平台必有可引用 before/after | 哈森闭环后：3 个数字 + 1 句客户证言 |
| B4 | **定价与交付边界** | 模块化在 05，缺打包价 | Starter-A：报工 Skill + **ERP write** Connector + 实施人天上限 |
| B5 | **渠道/ISV** | 金蝶/用友/鼎捷 SI 握客户 | 伙伴计划：伙伴做 Graph freeze，平台出 Harness+Skills |
| B6 | **竞争叙事** | 与灵基/YonClaw「企业 AI OS」撞车 | **「ERP Agent 管账；FactoryOS 管车间执行与跨系统报工」** |
| B7 | **续费产品** | 不能只卖实施 | 年费 = 对账 + 审计 + License + SLA（见 06） |

### 4.2 业务价值要顶级 — 还差什么

| # | 缺口 | 痛点来源 | 建议（Phase） |
|---|------|----------|---------------|
| V1 | **计件/工资协同 Skill** | 金蝶/迈斯：计件是买单核心 | **1.5**：报工 → 计件草案 → 财务 L2 确认 |
| V2 | **信息回灌现场** | Tulip：系统须把数据还给工人 | 工人端：本单进度、本工序是否落后（L0） |
| V3 | **主管班次指挥 Skill** | MES 采用靠主管每天用 | **2**：滞后预警 + 批量确认/驳回报工 |

## 五、与 MCP / Skills / Harness 的融合分配（纳入平台架构）

> 详见上轮沟通；此处作为差距补齐的**技术分配原则**，写入本审计文档以备 Phase 0 ADR 引用。

```text
Layer 0  可信内核（保留）     Graph + Rule + DSL + Audit + Revert
Layer 1  MCP Gateway          稳定动词 + Registry → Connector
Layer 2  Skills                场景剧本（报工/计件/主管）→ 绑定 Graph Pack
Layer 3  Harness Loop          感知→授权→沙箱→确认→执行→验证→记录
Layer 4  Evolution（后期）     仅优化 Skill 提示/步骤，不改 frozen Graph/Rule
```

| 层次 | 灵活性 | 商业价值 | Phase |
|------|--------|----------|-------|
| MCP | 新 ERP = 新 Registry | Connector Pack、伙伴生态 | P1 内部 / P2 开放 |
| Skills | 新场景 = 新 Skill | Graph/Capability Pack | P1 内置 2–3 个 |
| Harness | 换模型不改执行链 | Enterprise 治理、续费 | P1 必须 |
| Evolution | 越用越准 | 增值服务 | P3+ |

## 八、顶级平台验收标准（12 条）

当以下全部满足时，可认定垂直方案在**商业、业务、可落地**三维达「顶级楔子平台」：

1. 工人钉钉 **30 秒内**完成报工，无需纸质手册  
2. 主管 **24h 内**可 revert，**Legacy（ERP/MES）** 数量一致  
3. **计件工资草案**自动生成，财务仅确认  
4. 对账 Job **每日零 drift** 或自动告警  
5. 第二家同类型客户 **≤8 周**上线（import 模板）  
6. 客户签 **年费**（非纯一次性项目）  
7. 公开 **before/after** 三指标案例  
8. Graph/Rule 变更 **全程可审计**  
9. **Shadow 模式**为标准交付步骤  
10. 财务/IT 认可 **数据边界**（06）并签字  
11. 客户能说出 **与金蝶/用友 AI 的差异**  
12. 第 3 家交付 **不依赖创始人**个人英雄主义

## 九、审计结论（兜底确认后）

### 9.1 确认成立的判断

1. 制造业核心矛盾是 **Execution Gap + 数据孤岛 + 人的采用失败**，不是缺一个聊天 AI。  
2. **Overlay + Graph/Rule/DSL/Revert** 是正确且与全球趋势一致的工程路径。  
3. **报工楔子** 是正确首战；但鞋服客户买单常为 **报工+计件+透明**，需补 V1。  
4. **65%/70–80%** 类数据支持「先 shadow、先人在回路、先可复制」策略。  
5. 冲顶级缺的是 **产品化交付与 ROI 证明**，不是再堆概念层。

### 9.2 需自家验证、不宜外引的数据

- 扫码计件「50% 人力节省」「1% 错误率」等——仅作行业参考，**哈森案例须用实测**。  
- 「73% 规上服装厂已自动化计件」（i人事）——口径不明，不写入对外材料。

### 9.3 一句话

> **方向与架构经多源印证可靠；要三维顶级，请把「鞋服报工+计件+对账」做成可 export 的闭环产品，并用哈森数据证明 ROI——其余能力按本表 B/V/I 清单分期补齐。**
