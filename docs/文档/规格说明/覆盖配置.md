## 1. 为什么需要 Override

制造业 D2 扩展中约 **30%** 为真·一次性差异（08：≥70% 须 Pack 化）。剩余差异来自：

- 同一 Graph，**产线 A** 多一道确认节点
- **设备** 扫码格式不同
- **Connector** 字段映射因 MES 版本不同
- **Rule** 阈值因厂规不同

**Whale Tax 防线**：这些差异 **不得** 修改 `execution_service` / Platform-L0 源码；走 **Override 文档** 在运行时合并。

---

## 2. 三级 Override 模型

```text
Platform Override   →  全租户默认值（极少，仅平台级 feature flag）
Tenant Override     →  工厂级（默认层）
Scope Override      →  产线 / 设备 / 工序 / Connector 实例级
```

| 级别 | scope_type | 典型内容 | 谁可编辑 |
|------|------------|----------|----------|
| **Platform** | `platform` | 全局 DSL 开关、Shadow 默认 | 平台团队 + ADR |
| **Tenant** | `tenant` | 厂规阈值、默认 IM、对账 cron | 实施 + 工厂 IT |
| **Scope** | `line` / `equipment` / `workstation` / `connector_instance` | 节点分支、字段 map、Rule 参数 | 实施（D2 范围） |

**优先级**（合并时）：`Scope > Tenant > Platform > Pack 默认值`

## 3. Override 文档 Schema

**真源**：[OverrideDocument.schema.json](../数据结构/OverrideDocument.schema.json)（OpenAPI `$ref` 同文件）。

TypeScript 等价（摘要）：

```typescript
interface OverrideDocument {
  override_id: string;
  tenant_id: string;
  scope_type: 'platform' | 'tenant' | 'line' | 'equipment' | 'workstation' | 'connector_instance';
  scope_id?: string | null;
  version: string;
  status: 'draft' | 'active' | 'archived';
  patches: OverridePatch[];
  metadata: {
    created_by: string;
    approved_by?: string | null;
    approved_at?: string | null;
    d2_ref?: string | null;
  };
  created_at: string;
  updated_at: string;
}

interface OverridePatch {
  target: OverrideTarget;
  op: 'merge' | 'replace' | 'remove';
  value: unknown;
}

type OverrideTarget =
  | { kind: 'graph'; path: string }
  | { kind: 'rule'; rule_id: string; path: string }
  | { kind: 'dsl'; action_id: string; path: string }
  | { kind: 'connector'; registry_key: string; path: string }
  | { kind: 'skill'; module_id: string; path: string }
  | { kind: 'feature'; key: string };
```

**HTTP**（OpenAPI v1.1.1）：`GET/POST /v1/overrides` · `GET/PUT /v1/overrides/{overrideId}` · `POST .../activate` · `POST .../archive`。

**禁止的 Override target**：

- `execution_service` 内部逻辑路径
- 绕过 Rule Engine / 关闭 Audit
- 将 `graph.status` 改为非 frozen
- Agent 直写 Connector 的开关

违反 → 校验拒绝 `403 OVERRIDE_FORBIDDEN`。

## 4. 合并算法（运行时）

```text
1. 加载 frozen Graph G0（checksum 校验）
2. 收集 applicable overrides：
     platform active
   + tenant active for tenant_id
   + scope active where scope_id ∈ 当前执行上下文
3. 按 priority 排序：scope 深度优先（equipment > line > tenant > platform）
4. 依次 apply patches（RFC 7386 merge 语义）
5. 合并结果 G' 仅用于本次执行；**不**写回 Graph 存储
6. Rule Engine 对 G' + RuleSet' 求值
```

**Graph 版本**：Override **不**产生新 Graph 版本；若变更影响链路语义（增删节点），须 **clone Graph → 新版本 freeze**（业务图谱规格），Override 仅调参数。

## 5. 与 D1 / D2 的边界

| 场景 | 做法 |
|------|------|
| D1 通病五项 | **禁止** Scope Override；仅 Tenant 级 IM/Connector endpoint |
| D2 计件规则差异 | 优先新 Skill Pack；阈值用 Tenant Override |
| 产线多分支 | Scope Override 或新 Graph 版本（二选一，实施评估） |
| 新 MES 字段 | Connector Override mapping；≥2 厂相同 → conn Pack 升级 |
| 客户要求改内核 | **拒绝**；走 ADR 或新 Pack |

## 7. License 与 Pack 关系

- Override **不是**独立售卖 SKU 的首选形式；收入层 **L3**（09）
- `override-scope` Pack 表示「启用 Scope Override 能力」；具体文档按租户存储
- Tenant License 字段 `overrides: string[]` 列出 active override_id（模块包规格）

---

## 8. Implementation Package 中的 Override

export/import 时包含全部 `active` Override 文档（脱敏后）。import 到新 tenant 时：

1.  remap `tenant_id`
2.  remap `scope_id`（若产线编码不同，实施手动映射表）
3.  **不** import `platform` 级 Override

## 9. 验收（AC 摘要）

| ID | 条件 |
|----|------|
| OV-01 | Tenant Override 改变 Rule 参数，不改 Graph checksum |
