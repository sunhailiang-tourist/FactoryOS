# 过渡架构：四态演进模型

> 版本：v0.1.0 | 日期：2026-06-16  
> 用途：避免「一步微服务」或「永远单厂 hardcode」；与 [09-三年商业与技术路线图](../../准备/2026-06-16/09-三年商业与技术路线图.md) 对齐  
> 原则：**当前态可运行，下一态可演进，不提前支付分布式税**

---

## 1. 四态总览

```text
Baseline (Phase 0–1)  →  Y1 Target  →  Y2 Target  →  Y3 Target
单 tenant Monolith       多 tenant 列   License DB    可选拆服务 + 计量
```

| 态 | 时间 | 部署 | 租户 | License | Connector | 前端 |
|----|------|------|------|---------|-----------|------|
| **Baseline** | Phase 0–1 | 1× `server/api` | 单 tenant | env / JSON 全开 | Path A/B mock | web-admin 骨架 |
| **Y1 Target** | Phase 1–2 | 1× Monolith | 2–3 灯塔 | JSON 文件 | +Mock 认证 | H5 + admin |
| **Y2 Target** | Phase 2–3 | 1× Monolith | 5–10 | PostgreSQL | WMS/QMS/OA 读 | Console MVP |
| **Y3 Target** | Phase 4 | Monolith + 可选 2 服务 | 10+ / 伙伴 | + metering | 认证目录 | 多入口 |

> **MCP**：Y1 末内部 GA、Y2 对外（ADR-004，废止本文档旧「Y3 MCP 开放」单一表述）。

## 2. Baseline（Phase 0–1，Coding 启动态）

### 架构

```text
server/api (FastAPI + Uvicorn)
  ├── graph_service
  ├── rule_engine
  ├── execution_service
  ├── audit_service
  ├── agent_orchestrator   # LangGraph + LiteLLM
  ├── connector_sdk        # httpx; mes, erp, mock
  └── license_service      # stub: always allow 或静态 JSON

PostgreSQL 16 — 单 schema，tenant_id 列存在 default 'default'
```

### 必须做

- 所有表含 `tenant_id`（默认 `'default'`）
- `license-service` 接口定义 + stub 实现
- Pack ID 常量化（shared_contracts/errors）
- Connector Registry 10 类 enum

### 禁止做

- Kafka / 多服务部署
- 按客户 hardcode 分支
- 无 Pack 的功能开关

---

## 3. Y1 Target（灯塔多厂，Phase 1 末–Phase 2 初）

### 变更

| 项 | Baseline → Y1 |
|----|---------------|
| tenant | 2–3 行 tenant 表；middleware 注入 `tenant_id` |
| License | 每 tenant 一个 JSON 文件或 `tenant_licenses` 表 |
| Graph | `tenants/{id}/graphs/*.frozen.json` |
| Override | Tenant 级 connector config only |
| Package | export/import CLI 或 admin API |

### 部署

仍 **单 Monolith**；可按 tenant 逻辑隔离，物理仍单库（schema 或 row-level）。

### Gate

- 灯塔 B import Package 成功
- License 403 单测通过

---

## 4. Y2 Target（Pack 飞轮，Phase 2–3）

### 变更

| 项 | Y1 → Y2 |
|----|---------|
| Pack Registry | DB 表 + `文档/商业/packs/*.json` 同步 |
| Override | 完整 merge 引擎 + Scope |
| Connector | WMS/QMS/OA stub→impl |
| Console | Graph freeze + Pack 视图 + Override 编辑 |
| 队列 | Redis/NATS 用于对账 Job（可选） |

### 部署

仍 Monolith 为主；**不拆** `execution_service`（写路径唯一性 ADR-001）。

### Gate

- ≥10 Pack M3+
- D2 入库率 ≥70%

---

## 5. Y3 Target（规模化，Phase 4）

### 可选拆分（仅当 Monolith 成为瓶颈）

| 服务 | 拆分条件 | 职责 |
|------|----------|------|
| `license-service` | License 校验 >20% CPU 或 独立发布周期 | Pack、Override、metering |
| `connector-host` | Connector 进程隔离（客户 VPN） | 仅 `connector_sdk` |

**写路径不变**：即使拆 connector-host，`execution_service` 仍是唯一 orchestrator。

### 新增

- Usage metering（写次数 / Connector 调用）
- MCP Registry 对外（Platform-L1）
- Partner Pack 导入 + 签名

---

## 6. 数据迁移原则

1. **只加列、不加破坏性迁移**（tenant_id 从 day1 存在）
2. License 从 JSON → DB：import 脚本，非手工
3. Graph frozen 实例 **永不** 跨 tenant 自动共享；仅通过 Implementation Package 显式 import

---

## 7. 反模式（禁止）

| 反模式 | 为何 |
|--------|------|
| Phase 1 上 K8s 多集群 | 运维税 > 业务价值 |
| 每客户一个 repo fork | 飞轮死亡 |
| 先微服务后补 Pack | SKU 与部署单元错位 |
| 去掉 tenant_id「以后再加」 | Y1 全表迁移 |

## 9. 规模阶梯（与 ADR-007 对齐）

| 态 | 对应 ADR-007 | 租户量级 |
|----|--------------|----------|
| Baseline～Y3 | **S0** | 1～10+ |
| Y2～Y4 能力目标 | **S1 百级** | 10～10² |
| Y4～Y5 | **S2 千级** | 10²～10³ |
| Y5+ | **S3** | 10³+ |

> Y1～Y3 四态（§1～§5）为 **商业与交付节奏**；S0～S3 为 **基础设施能力阶梯**。二者并存，以 ADR-007 为规模真源。
