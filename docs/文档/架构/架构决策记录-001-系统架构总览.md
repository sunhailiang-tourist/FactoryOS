## 背景

FactoryOS 是制造业 **AI 执行平台地基**：Overlay 现有 ERP/MES，不替换账本。

**双极架构**（产品与技术成对）：

| 极 | 承诺 | 主要层 |
|----|------|--------|
| **终端智能极** | 0 智、多模态（说/拍/扫）；极致应用体验 | Platform-L3 Harness + 感知入口 + Platform-L2 Skills |
| **内核门禁极** | frozen Graph、Rule、DSL、Audit、Revert；极致可靠 | Platform-L0 |

智能极负责 **让人脱离老系统的高心智操作**；门禁极负责 **敢写库、可追责、可回滚**。详见 [感知与多模态入口](../规格说明/感知与多模态入口.md)、[Harness终端体验](../规格说明/Harness终端体验.md)。

服务端 **全部采用 Python**（AI 开发与 LangGraph 生态）；前端 **React + TypeScript**（Phase 1 后半）。

当前阶段：**只建底座**，不接具体工厂业务 Graph。

## 决策

### 1. 总体形态：Modular Monolith

- 单 repo、单 deployable（`src/server/api` · import: `server.api`）
- 按 Python package 划分模块边界，Phase 3 后可拆微服务
- 禁止 Phase 1 引入分布式复杂度（无 Kafka 集群、无多服务部署）
- 包管理：**uv** workspace（或 Poetry workspace）

### 2. 技术栈（Phase 0–1 锁定）

#### 2.1 后端（Python）

| 层 | 选型 | 说明 |
|----|------|------|
| 语言 | **Python 3.12+** | 服务端唯一语言 |
| Web 框架 | **FastAPI** | 原生 async；OpenAPI 3.1 自动生成 |
| ASGI 服务器 | **Uvicorn** | 开发单进程；生产多 worker |
| 数据库 | **PostgreSQL 16** | Data-L1 运行数据 |
| ORM / 访问 | **SQLAlchemy 2.0 async** + **asyncpg** | 全表 `tenant_id` |
| HTTP 客户端 | **httpx**（async） | Connector 调 Legacy 唯一客户端 |
| 契约校验 | **Pydantic v2** + **jsonschema** | Pydantic 对齐 `文档/数据结构/` JSON Schema |
| 日志 | **structlog** | JSON 行，便于 Audit |
| 测试 | **pytest** + **pytest-asyncio** + **httpx** ASGI | 对照验收 BASE-001 |
| 类型检查 | **pyright** 或 **mypy** | os_core 严格模式 |
| 异步任务（Phase 2+） | **Redis** + **ARQ** | 对账 Job、Saga 补偿；Phase 1 可不引入 |

**选型理由**：FactoryOS 主负载为 MES/ERP/LLM **I/O 等待**；FastAPI asyncio 栈在高并发连接场景表现优于同步 Python 框架，且 AI 库生态完整。

#### 2.2 AI（仅 Platform-L2）

| 层 | 选型 | 落位 |
|----|------|------|
| Agent 编排 | **LangGraph** | `os_core/agent_orchestrator/` — 薄 FSM，非 open-ended Agent |
| 模型路由 | **LiteLLM** | 同一接口切换 OpenAI / Anthropic / 国内模型等 |
| 结构化输出 | **Pydantic** + LangGraph structured output | 输出 **DSL 计划 JSON**，禁止自然语言直写 Legacy |

**LLM 禁止出现在**：`graph_service`、`rule_engine`、`execution_service`、`audit_service`、`connector_sdk`（见 ADR-002）。

#### 2.3 前端（Phase 1 后半）

| 层 | 选型 |
|----|------|
| 框架 | **React 18** + **TypeScript** |
| 构建 | **Vite** |
| 应用 | `src/apps/web-admin`（PC）、`src/apps/h5-worker`（钉钉/企微 H5 壳） |

> **路径勘误（2026-06-26）**：代码根已 Consolidate 至 `src/`。HTTP 入口物理路径 `src/server/api/`（import: `server.api`）；内核 `src/server/os_core/`（import: `os_core`）；客户端 `src/apps/`；integration 镜像 `src/integration/`。

### 3. 模块边界与架构层映射

```text
src/server/api/              # FastAPI：application/ · router/ · config/ · modules/（import: server.api）
src/server/os_core/          # 10 内核模块 · registry.py（import: os_core.*）
  shared_contracts/
  platform_registry/         # ADR-008 Contract/Pack/Tenant DB
  graph_service/
  rule_engine/
  execution_service/
  audit_service/
  agent_orchestrator/
  connector_sdk/
  license_service/
  mcp_gateway/
src/server/db/migrations/    # Alembic
src/server/edge-agent/       # 私网 PoC
src/apps/
  web-admin/
  h5-worker/
src/integration/             # export/fixture 镜像 · registry.py
```

| 架构图层 | Python 模块 |
|----------|---------------|
| Platform-L3 Harness | `src/server/api` 路由 + 前端 |
| Platform-L2 Skills | `agent_orchestrator` + `license_service` |
| Platform-L0 信任内核 | `graph_service` · `rule_engine` · `execution_service` · `audit_service` |
| Platform-L1 Connector | `connector_sdk` + Blueprint Runtime + **`mcp_gateway`**；**MCP Y1 末内部 GA，Y2 对外**（ADR-004） |

#### 3.1 AgentModule 契约（P1 defer HTTP）

- **Schema**：[`代理模块.schema.json`](../数据结构/代理模块.schema.json) — Skill FSM 定义，随 Implementation Package 导入  
- **Core 1.0 / W1–W8**：**仅 filesystem**（`integration/packs/*/agent_modules/`）；**无** `/v1/agent/modules` HTTP CRUD  
- **P1+**：若需 Studio 在线编辑，另开 ADR 增补 OpenAPI；不得 W1 临时发明路径

### 4. 写路径唯一性

```text
Agent（LangGraph）→ DSL 计划 → Rule Engine → Execution Service → Connector（httpx）→ Legacy
```

任何绕过 `execution_service` 的写操作 **禁止**。

### 5. 平台分层（术语，避免与 Module Pack / Data Layer 混淆）

| 名称 | 含义 |
|------|------|
| **Platform-L0** | 信任内核：Graph, Rule, DSL, Audit, Revert |
| **Platform-L1** | MCP Registry + 稳定动词 |
| **Platform-L2** | Skills（场景剧本，业务接入后挂载） |
| **Platform-L3** | Harness Loop（感知→授权→沙箱→确认→执行→验证→记录） |
| **Data-L0~L3** | 见 [06-数据边界与部署策略](../../准备/2026-06-16/06-数据边界与部署策略.md) |
| **Pack-L1~L3** | Graph / Capability / Connector Pack，见 05 |

### 5.1 双极不可偏废

- **禁止** 只做 L0 门禁而无工人/主管可感的 H5 多模态（「后台正确但没人用」）
- **禁止** 只做聊天入口而无 Rule/Graph/Execution 门禁（「好用但不敢写库」）
- D1 结案须同时满足 [AC-BASE-001](../验收/验收用例-BASE-001-平台底座.md) 与 [AC-UX-001](../验收/验收用例-UX-001-终端体验与多模态.md) P0

### 6. 与业务的关系

- 底座交付 **机制**，不交付 **具体 frozen Graph**
- 第一个 vertical Skill（如报工）在工厂 Graph workshop 后挂载至 `agent_orchestrator`
- `文档/业务图谱/` 下仅保留 **draft 模板**，frozen 实例按租户存放（Phase 1 起 `tenants/{id}/graphs/`）

### 7. 部署（Baseline）

```text
生产：Uvicorn workers（≈ CPU 核数）+ Nginx 反向代理
开发：`uv run uvicorn server.api.main:app --reload`
数据：PostgreSQL 16；Phase 2 可选 Redis
```

## 后果

- 正面：Python AI 栈统一；FastAPI 与 OpenAPI/验收流程一致；L0 内核与 LLM 物理隔离清晰
- 负面：需维护 Pydantic 与 JSON Schema 双轨对齐；ADR-001 修订后须以本文件为准（取代原 TypeScript 决策）
- 前端仍为独立 TS 项目，需维护 OpenAPI 生成的客户端或手写 API 类型
