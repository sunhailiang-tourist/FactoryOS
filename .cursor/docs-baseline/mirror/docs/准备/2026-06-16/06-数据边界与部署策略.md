# 数据边界与部署策略

> 整理日期：2026-06-16  
> 问题来源：方案输出阶段——是否有必要做客户的业务数据跟踪，还是只做客户部署？  
> 关联文档：[04-工厂实施手册](./04-工厂实施手册.md)、[05-模块化与商业化方案](./05-模块化与商业化方案.md)、[ADR-001](../../文档/架构/架构决策记录-001-系统架构总览.md)

---

## 一、核心问题

作为 FactoryOS 方案输出方，我们需要明确：

1. 是否要为客户建立**全量业务数据跟踪/托管平台**？  
2. 还是仅提供 **Connector 部署 + Agent 上线** 后离场？  
3. 两者边界在哪里，如何写入对外方案与合同？

---

## 二、结论（方案输出口径）

> **不必、也不应默认做「客户全量业务数据跟踪平台」；必须做「部署 + 运行跟踪 + 对账」；业务事实仍以客户 ERP/MES 为账本，FactoryOS 只保留运行所需的最小数据。**

| 模式 | 是否采用 |
|------|----------|
| 纯部署实施（装完离场） | ❌ 不够——无法审计、回滚、对账、续费 |
| 部署 + 运行平台（推荐） | ✅ Overlay 标准模式 |
| 部署 + 全量业务数据中台 | ❌ 非默认；仅产品 B 或单独高价项目 |

**对外标准话术**：

- ✅ 「部署 FactoryOS，接管报工/查询入口；全程可审计、可回滚、与 ERP 对账」  
- ✅ 「数据权威仍在您的 ERP/MES」  
- ❌ 「我们将您全厂业务数据接到云平台统一跟踪」（除非单独立项 Layer 3）

---

## 三、概念澄清：「业务数据跟踪」指什么

| 类型 | 内容 | 示例 | 默认策略 |
|------|------|------|----------|
| **A. 执行跟踪** | FactoryOS 自身产生的运行数据 | ExecutionRecord、审计日志、Graph/Rule 版本、对账结果 | **必做** |
| **B. 业务镜像/缓存** | 为 Agent/Graph/revert 服务的最小业务快照 | 报工前后 qty、工单状态缓存、孪生节点引用 | **按需、最小化** |
| **C. 业务数据池** | ERP/MES 全量或长期同步至 FactoryOS | 全厂订单/库存/财务库复制、跨客户汇总 | **默认不做** |

ChatGPT 原对话中的「生产数据跟踪」应理解为 **A + B**（Agent 帮用户查/报/追溯 + 执行留痕），而非 **C**（平台级业务数仓）。  
「形成数据池」仅适用于 **C 类无 ERP 客户 / 产品 B**，不纳入 Overlay 标准包。

## 四、四层数据策略

```text
Layer 0  客户账本（权威）       ERP / MES / WMS / OA  — 客户拥有，FactoryOS 不替代
Layer 1  运行数据（必做）       Graph、Rule、Audit、ExecutionRecord、对账日志
Layer 2  业务缓存（按需）       查询加速、报工快照、Graph 节点状态 — 最小字段、可 TTL
Layer 3  分析/数据池（选配）   客户明示采购；建议私有化 — 非 Overlay 默认
```

### Layer 0：客户账本（权威源）

- 订单、库存、财务、报工单据的**事实记录**在 ERP/MES。  
- FactoryOS **不成为**第二套账本（Overlay 模式下）。  
- 与 ADR-001 / [16-OS核心基座架构设计方案](./16-OS核心基座架构设计方案.md) 中 Data-L0「客户账本」定义一致。

### Layer 1：运行数据（必做，与部署绑定）

| 数据类 | 用途 | 留存建议 |
|--------|------|----------|
| Graph / Rule 版本与 freeze 签字 | 链路锁定 | 长期 |
| ExecutionRecord + before/after 快照 | 回滚、对账 | revert 窗口 + 合规期（如 1–3 年） |
| Audit Log（append-only） | 企业审计 | 长期 |
| 对账结果（一致/drift） | 运维、验收 | 长期 |
| 租户配置、License、Connector 状态 | 商业化、实施 | 合同周期 |
| 实施/UAT 签字记录 | 责任边界 | 长期 |

**说明**：这是产品内核，不是「抢夺客户业务数据」，而是 **FactoryOS 运行证明**。

### Layer 2：业务缓存（按需）

原则：**能读 ERP 就不落库；要落库就 TTL + 最小字段。**

| 场景 | 可缓存 | 不可做 |
|------|--------|--------|
| 报工 | 工单当前量、最近 N 笔报工 | 全量 MES 表同步 |
| 只读查询 Agent | 短期查询结果 TTL | 替代 ERP 为主数据源 |
| Graph 孪生 | 实体引用关系 | ERP 全表拷贝 |

### Layer 3：分析/数据池（选配模块）

适用：

- C 类无 ERP 客户（产品 B）  
- 客户明确要求私有化「生产看板 / 历史分析」  
- 集团多厂汇总（建议**集团私有云**，非厂商 SaaS 混存）

不适用：Overlay 标准包默认承诺。

## 五、三种商业模式对比

| 模式 | 做什么 | 收入 | 风险 |
|------|--------|------|------|
| **纯部署商** | Connector、Graph freeze、培训、离场 | 一次性 | 难续费、难证明价值 |
| **部署 + 运行平台（推荐）** | 上述 + Layer 1 + 对账/审计/模块 License | 实施费 + 年费 | 可控 |
| **部署 + 业务数据中台** | 全量同步 + 数仓 + 分析 | 项目制 | 责任大、偏离 Overlay |

**方案应销售「部署 + 运行平台」**，产品名可表述为：

> **FactoryOS 部署与运行服务**

## 六、按客户类型的数据策略

| 客户类型 | 数据策略 | 部署形态 |
|----------|----------|----------|
| **A 类**（有 ERP） | 账本在 ERP；FactoryOS 仅 Layer 1 + 必要 Layer 2 | 私有化 / 客户独立 tenant |
| **B 类**（ERP + Excel） | 不以 FactoryOS 为唯一数据源；Excel 只读接入，不默认全量入库 | 同上 |
| **C 类**（无系统） | 可谈 Layer 3 数据池（产品 B） | 单独报价、单独合同 |

## 七、与 ChatGPT 原对话的对应关系

| 原对话表述 | 本策略界定 |
|------------|------------|
| 「打通…生产数据的跟踪」 | Layer 1 执行跟踪 + Layer 2 查询/报工，**非** Layer 3 |
| 「不动他原有的数据」 | Layer 0 权威在 ERP，Overlay 只读/经确认写入 |
| 「形成他们的数据池」 | 仅 C 类 / 产品 B / Layer 3 选配 |
| 「并行…读数据…确认落库」 | Layer 1 ExecutionRecord + Connector 写 API |

## 八、与模块化、实施的关系

| 模块/环节 | 涉及数据层 |
|-----------|------------|
| Graph / Rule / Audit | Layer 1（必做） |
| 报工 L2 + Revert | Layer 1 + 最小 Layer 2 |
| 生产查询 L0 | 以 Connector 读 Layer 0 为主，Layer 2 可选缓存 |
| 实施对账、UAT | Layer 1 对账日志 |
| 「行业数据洞察」/ 跨客户分析 | **默认不做**（涉及跨租户数据） |

详见 [05-模块化与商业化方案](./05-模块化与商业化方案.md)、[04-工厂实施手册](./04-工厂实施手册.md)。

## 九、方案与合同话术

### 建议写入方案书

```text
1. 业务数据权威系统：客户现有 ERP/MES/WMS。
2. FactoryOS 存储：业务流程定义（Graph）、规则、执行与审计日志、回滚所需最小快照。
3. FactoryOS 不默认托管客户全量业务数据，不做跨客户数据汇总。
4. 数据部署：默认私有化或客户专属 tenant，数据不出厂（除非客户书面同意其他模式）。
5. 可选模块：生产分析看板、历史数据服务（Layer 3）单独报价。
```

### 建议避免承诺

- 「我们统一跟踪贵厂全部业务数据」  
- 「数据存放在 FactoryOS 云端供我方分析」  
- 「替代 ERP 成为唯一数据源」（Overlay 模式下）

## 十、工程实现要点（摘要）

与 [03-锁定实施策略](./03-锁定实施策略.md)、[16-OS核心基座架构设计方案](./16-OS核心基座架构设计方案.md) 一致：

- **L2 数据采集层**：结构化采集表结构/接口/样本单据，**不让 AI 猜**；不等于全量入库。  
- **ExecutionRecord**：每次写操作的锚点，含 `erp_refs`、before/after 快照。  
- **对账 Job**：FactoryOS 记录 ↔ ERP/MES 逐笔核对，drift 告警。  
- **租户隔离**：多厂数据逻辑隔离；厂商侧不混用客户业务数据做训练/分析（除非合同授权）。

**云基础设施定版**（采购、安全组、扩容路径）：见 [10-阿里云基础设施定版方案](./10-阿里云基础设施定版方案.md)。  
**软硬件与 API 总账**（Cursor、研发期 LLM/多模态，不含人力）：见 [12-软硬件与API成本框架](./12-软硬件与API成本框架.md)。  
**研发资源与云环境用途清单**（测试/生产细分、全流程对照）：见 [15-研发资源与云环境用途清单](./15-研发资源与云环境用途清单.md)。

## 十一、决策检查清单

对外方案定稿前确认：

- [ ] 是否明确 Layer 0 权威在 ERP/MES？  
- [ ] 是否列出 Layer 1 必存数据类型？  
- [ ] 是否说明 Layer 2 缓存策略（TTL、最小字段）？  
- [ ] Layer 3 是否仅作为选配、单独报价？  
- [ ] 部署形态（私有化/tenant）是否写清？  
- [ ] 是否避免「全量业务数据托管」的模糊表述？

---

## 十二、一句话

> **方案输出主打「部署 + 运行保障」：执行跟踪、审计、对账、回滚；不做默认的「客户全量业务数据托管」。业务事实在 ERP/MES；FactoryOS 存 Graph、规则、执行记录与 revert 所需最小快照。无 ERP 或客户采购分析模块时，再单独立项 Layer 3。**
