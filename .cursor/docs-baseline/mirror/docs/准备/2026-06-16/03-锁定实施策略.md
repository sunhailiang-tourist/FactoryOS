## 一、策略目标

> **战略上下文**（[08](./08-平台战略与两次交付模型.md)）：工程 Phase 服务于「平台地基 + 两次客户交付」——Phase 1 完成后方可启动灯塔厂 **第一次交付（通病包 D1）**；工厂特异能力进 **第二次交付（扩展包 D2）**，≥70% 沉淀为 Pack。

在写第一行代码前，锁定三件事：

| 锁定项 | 内容 | 不锁定的后果 |
|--------|------|--------------|
| **北极星场景** | 哈森 1 条产线 × 工人报工 Agent × **ERP（写+读）+ 钉钉**（无独立 MES） | AI 生成「万能 OS」，永远做不完 |
| **架构形态** | Modular Monolith + 清晰模块边界 | 代码散乱、难测、难回滚 |
| **工程红线** | Graph 冻结 + DSL + Rule Engine + ExecutionRecord + Audit | Agent 直调 ERP，后期无法补救 |

**AI Coding 前提**：AI 擅长在**明确契约**下写模块，不擅长在模糊愿景下设计整个 OS。先锁 Schema / ADR / 验收用例，再 Coding。

## 二、锁定的架构设计

### 2.1 总体形态：一个核心 + 三个域 + 一个入口

```text
┌─────────────────────────────────────────────────────────┐
│  factoryos-web / dingtalk-h5 / wecom-h5   （多端入口）    │
└───────────────────────────┬─────────────────────────────┘
                            │
┌───────────────────────────▼─────────────────────────────┐
│  agent-orchestrator     意图 → 选 Module → 生成 DSL 计划   │
│  （无 ERP 写权限，只产出 DSL 请求）                          │
└───────────────────────────┬─────────────────────────────┘
                            │
        ┌───────────────────┼───────────────────┐
        ▼                   ▼                   ▼
┌───────────────┐  ┌───────────────┐  ┌───────────────────┐
│ graph_service │  │ rule_engine   │  │ execution_service │
│ 版本/冻结/查询 │  │ 权限/条件/分级 │  │ DSL执行/Saga/Undo │
└───────┬───────┘  └───────┬───────┘  └─────────┬─────────┘
        │                  │                    │
        └──────────────────┼────────────────────┘
                           ▼
                 ┌──────────────────┐
                 │ audit-service    │  append-only
                 └────────┬─────────┘
                          ▼
                 ┌──────────────────┐
                 │ connector_sdk    │  ERP / MES 适配
                 └────────┬─────────┘
                          ▼
                    Legacy 系统
```

**Phase 1**：上述逻辑放在**一个 repo、一个 deployable**（Modular Monolith），按 package 分模块。微服务放到 Phase 3（≥3 家外部客户）后再拆。

### 2.2 技术选型（锁定）

> **以 [架构决策记录 001](../../文档/架构/架构决策记录-001-系统架构总览.md) §2 为准**（2026-06-16 修订：Python 后端 + React/TS 前端）。

| 层 | 选型 | 理由 |
|----|------|------|
| 后端语言 | **Python 3.12+** | AI 开发、LangGraph 生态 |
| API | **FastAPI** + OpenAPI 3.1 | async I/O；规格先行 |
| 数据库 | **PostgreSQL 16** + SQLAlchemy async | Graph、ExecutionRecord、Audit |
| Connector | **httpx** async | 调 MES/ERP |
| 队列（Phase 2+） | Redis + ARQ | 对账、异步补偿 |
| Agent 编排 | **LangGraph**（薄 FSM） | 固定 Skill，不用 open-ended agent |
| LLM | **LiteLLM** | 可切换 Provider；仅 `agent_orchestrator` |
| 前端 MVP | **React + TypeScript** + Vite；钉钉/企微 H5 | 工人端先 H5 |

**硬约束**：所有写 ERP/MES 的路径**只有** `execution_service` 可走；CI / Cursor rules 强制检查。

### 2.3 核心数据契约（Coding 前先审定）

以下 5 份 JSON Schema 为系统「宪法」，人工审定后再生成代码：

| Schema | 职责 |
|--------|------|
| `BusinessGraph` | nodes / edges / status / version / checksum |
| `RuleSet` | 绑定 graph_version；action + role + condition |
| `DSLAction` | 动作词汇表 + params schema + compensator |
| `ExecutionRecord` | exec_id / before / after / erp_refs / revert |
| `AgentModule` | allowed_dsl + workflow FSM |

存放路径（规划）：`文档/数据结构/`

### 2.4 MVP 相对 12 层的裁剪

| 12 层 | Phase 1 | Phase 1 做法 |
|-------|---------|--------------|
| L1 Connector | ✅ | **Path A**：ERP 写+读 + 钉钉；**Path B**：MES 写 + ERP 读；或 mock |
| L2 数据采集 | ✅ 部分 | 仅报工相关表/接口 |
| L3 业务发现 | ❌ | 人工 + AI 辅助画 Graph Draft |
| L4 人工确认 | ✅ | Admin UI freeze Graph |
| L5 数字孪生 | ⚠️ 极简 | Graph 仅含报工链路 5–8 节点 |
| L6 Rule Engine | ✅ MVP | 角色 + 3 条规则 |
| L7 DSL | ✅ | 4 个动作（见下） |
| L8 Agent | ✅ | 1 个 Module：`work_report_v1` |
| L9 Audit | ✅ | append-only |
| L10 Sandbox | ⚠️ | dry_run 影响报告 |
| L11 Rollback | ✅ | 仅 `WORK_REPORT_REVERT` |
| L12 Execute | ✅ | 经 Connector 写 **Legacy（ERP/MES，按 Path A/B/C）** |

**Phase 1 DSL 词汇表（锁定）**

```text
QUERY_WO
WORK_REPORT
WORK_REPORT_REVERT
QUERY_REPORT_HISTORY
```

## 三、工程红线（写入 ADR-002 与 Cursor rules）

| 禁止 | 必须 |
|------|------|
| Agent 直接写 ERP/MES 数据库 | Agent → DSL → Rule Engine → API → Legacy |
| AI 100% 自动推导业务链路 | AI 草稿 + 人工确认 + Graph 冻结 |
| 无版本在线改 Graph | Graph V1.0 / V1.1 / V2.0 版本化 |
| 跳过 Rule Engine 执行 | 规则引擎不可绕过 |
| 生产环境直接试写 | dry_run / 沙箱 → 确认 → 生产 |
| 无 ExecutionRecord 的写操作 | 每次写操作可追溯、可补偿 |

**设计优先级（不可颠倒）**

```text
可靠性 > 安全性 > 可追溯 > 可回滚 > AI 能力
```

**核心投入顺序**（单轨依赖顺序；**冲刺模式下 A/B 双轨并行**，见 `14`）

```text
1. Business Graph          ← A 轨
2. Rule Engine             ← A 轨
3. Audit System            ← A 轨
4. Rollback Engine         ← A 轨
5. Agent Orchestrator      ← A 轨（与 B 轨感知/Harness 第 3 周集成）
6. 多模态交互              ← B 轨（与 1～4 并行，不等待全部完成）
```

## 五、AI Coding 方法论

### 5.1 开发循环

```text
1. 写模块 README + OpenAPI / JSON Schema（人审定）
2. 写 acceptance tests / contract tests（AI 生成，人审定）
3. AI 实现代码
4. CI 跑通
5. 进入下一模块
```

**禁止**：先让 AI scaffold 整个系统再补文档。

### 5.2 人 vs AI 分工

| 人锁定 | AI 生成 |
|--------|---------|
| 架构决策 ADR | CRUD、业务逻辑 |
| JSON Schema / OpenAPI | 单元测试、集成测试 |
| Graph v1.0 业务内容 | Connector 样板 |
| Rule 规则定义 | UI 组件 |
| 验收标准 | 文档、脚本 |
| Code Review 红线 | Refactor |

### 5.3 策略文档包（Coding 前必备）

| 文档 | 作用 |
|------|------|
| ADR-001 架构总览 | AI 上下文 |
| ADR-002 安全与执行红线 | Cursor rules 来源 |
| 业务图谱规格 | Graph 模型与状态机 |
| DSL规格 | 动作词汇表 + compensator |
| 执行规格 | ExecutionRecord + Saga + Undo |
| 连接器规格 | 适配器接口规范 |
| AC-BASE-001 | 平台底座验收（Graph/Rule/DSL/Execution） |
| AC-MVP-001 | 垂直报工场景验收（工厂 Graph frozen 后） |

## 七、组织节奏

```text
每周
  周一：锁定本周 1 个模块 spec + 验收标准
  周二–四：AI Coding + 测试
  周五：Demo + 对账 + 更新 ADR

每 Phase 结束
  哈森业务 walkthrough
  Graph 是否需 v1.x 升级
  再开下一 Phase
```

**禁止并行**：Graph 平台、ERP 全家桶、独立 App、Boss Agent 与 Phase 1 报工闭环同时进行。

---

## 八、三个 Gate 问题（每 Phase 必问）

1. **链路锁住了吗？** — 未 freeze 的 Graph 能否被执行？（必须不能）
2. **回滚证明了吗？** — 随机报工能否 revert 且与 **Legacy read-back（按 Path）** 一致？
3. **AI 生成可控吗？** — 删掉 AI 代码，spec + test 能否指导重写？

## 十、一句话策略

> **FactoryOS = 以 frozen Business Graph 为宪法、DSL+Rule Engine 为唯一写路径、ExecutionRecord 为回滚锚点的 Modular Monolith；全程 AI Coding，先锁 Schema/ADR/验收，Phase 1 用 6–8 周交付报工闭环，再扩展读分析与写能力。**
