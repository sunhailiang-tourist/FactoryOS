## 1. 背景

F1.5 后 Core 1.0 设计目标为 **Y3「10+ tenant / 10～20 Silver Connector」**（`过渡架构` §1、`17` §7.2）。业务追问：**百企、百 ERP、百 OA 是否优雅支撑？千级如何预设？**

全库 119 篇文档通读结论：**L0 写路径与 GIP 外置策略可扩展至千级**；**当前 Y1 单栈 Monolith 物理拓扑不能直扛百级**，须 **分级部署 + Cell + Connector 实例化**，且 **W1 起预埋路由/Outbox 字段**，避免 Y2 全表迁移。

**站外对齐**：[AWS 混合多租户 Tier/Cell/InfraGroup](https://aws.amazon.com/blogs/architecture/building-hybrid-multi-tenant-architecture-for-stateful-services-on-aws/)、[Cell-Based Architecture 2026](https://techbytes.app/posts/cell-based-infrastructure-deep-dive-2026-scale/)、[iPaaS+MCP 多租户指南](https://labs.strongstart.digital/multi-tenancy-guide-for-ipaas-mcp-architecture-notes)、[webMethods hub-spoke EKS](https://aws.amazon.com/blogs/containers/how-webmethods-ipaas-built-a-multi-tenant-saas-platform-on-amazon-eks/)。

## 2. 规模阶梯（Scale Ladder）— 单一真源

| 阶梯 | 租户量 | Connector 实例量（量级） | 部署形态 | 时间窗（能力，非承诺） |
|------|--------|---------------------------|----------|------------------------|
| **S0** | 1～10 | 10～50 | 单 Monolith + 单 RDS（`10` §二） | Baseline～Y3（现行） |
| **S1 百级** | 10～10² | 10²～10³ | **Pool** 多租户 + RLS + 读副本 + Outbox Worker | Y2～Y4 |
| **S2 千级** | 10²～10³ | 10³～10⁴ | **Cell** 数据面 + 共享控制面 + 按 tenant 限流 | Y4～Y5 |
| **S3 超大** | 10³+ | 10⁴+ | Cell _shuffle + 大客户 **Silo** 独立栈 | Y5+ |

**计数口径**：

- **tenant** = 计费与隔离边界（`tenant_id`）
- **factory / site** = `scope_id` 或 License `factory_count`（一 tenant 多厂）
- **Connector 实例** = `(tenant_id, pack_id, environment, instance_id)` — **不是** 100 种 ERP 内核

**废止歧义**：「100 ERP」= **100 个 ConnectorInstance**（多为同型 Blueprint 复制），**不是** 100 个 LegacySystem enum（仍 **10 类**，`连接器` §2）。

## 3. 三态部署模型（Pool / Bridge / Silo）

统一 `04`「每厂独立部署」与 `10`「同栈多 tenant_id」：

| 模式 | SKU | 隔离 | 适用 | 与 04 关系 |
|------|-----|------|------|------------|
| **Pool** | Community / Starter / Pro | 共享 App+DB + **RLS** + 应用层过滤 | S1 百级 SMB | 多厂 **逻辑** tenant |
| **Bridge** | Pro / Enterprise  mid | 共享 App + **独立 PG schema**（或 DB） | 合规 mid-market | 同 VPC 多 schema |
| **Silo** | Enterprise | **独立栈**（ECS+RDS+Redis+OSS） | 私有化 / 超大单客 | `04` §Step 1.1 **每厂独立部署** |

**决策 S7-01**：三种模式 **共用同一 L0 二进制**（`apps/api` 同 artifact）；差异在 **provision 模板 + `placement_tier`**，**禁止** fork Core 代码。

**决策 S7-02**：S0～S1 默认 **Pool**；Enterprise 合同 **Silo** 仍走 **Implementation Package import**（`08` Gate），不另建内核分支。

## 4. 控制面与数据面分离

```text
┌─ 控制面（全局共享，S2 起独立 deployable 可选）────────────────┐
│ Tenant Registry · Cell Router · License/Quota · CMV Registry   │
│ Studio · MCP OAuth · Pack 认证目录 · 全局 Audit 索引（只读）      │
└───────────────────────────┬──────────────────────────────────┘
                            │ 路由 tenant_id → cell_id
┌─ 数据面 Cell（S1 起逻辑分区，S2 起物理 Cell）──────────────────┐
│ apps/api Monolith · RDS（schema/实例）· Redis · OSS prefix      │
│ integration/catalog（只读镜像）· Edge Agent 会话               │
│ connector_worker（S2 可选进程池，仍经 execution_service 写）     │
└───────────────────────────────────────────────────────────────┘
                            │ HTTPS / WSS / PrivateLink
                     客户 ERP / MES / OA（Legacy Data-L0）
```

**决策 S7-03**：**写路径唯一性不变** — 即使 S2 拆 `connector_worker`，**仅** `execution_service` 可发起 L2 写（ADR-001、膨胀期 §3 禁止 extract L0 四模块）。

**决策 S7-04**：控制面 **不** 处理 L2 写流量；tenant  provisioning 与 Cell 扩容 **不得** 阻塞 execute 热路径（对齐 AWS/Azure 控制/数据面分离共识）。

## 5. Cell 架构与千级预设

### 5.1 Cell 定义

| 字段 | 含义 |
|------|------|
| `cell_id` | 数据面单元 ID，如 `cell-cn-east-01` |
| `tier` | `standard` \| `high_tps` \| `enterprise_dedicated` |
| `max_tenants` | 软上限（Standard **200**；Dedicated **1**） |
| `region` | 数据驻留 |

### 5.2 路由

```text
请求 → SLB/Nginx（Global Router）
     → 查 TenantRegistry(tenant_id) → cell_id + placement_tier
     → 转发至 Cell 入口（内网 DNS / PrivateLink）
```

**决策 S7-05**：**W1 预埋** `tenants.cell_id`（nullable，默认 `cell-default`）；S0 路由 **no-op**（单 Cell）。S2 启用路由 **不改** L0 API 契约。

**决策 S7-06**：Standard Cell **`max_tenants = 200`**（**审定确认**）；Enterprise Dedicated = **1 tenant/Cell**。扩 Cell 触发见 §12。

### 5.3 千级基础预设（W1 Coding 必做）

| 预埋项 | W1 实现 | S2 启用 |
|--------|---------|---------|
| `tenants.placement_tier` | enum，默认 `pool` | provision 时赋值 |
| `tenants.cell_id` | default `cell-default` | Router 生效 |
| 复合索引 `(tenant_id, …)` 全表 | ✅ Gate 0 | 分区表可选 |
| `outbox_events` 表 | 写入 stub / 同学进程 | Worker 消费 |
| `connector_instances` 表 | schema only | Runtime 绑定 |
| `tenant_quotas` 表 | schema + stub deny | 限流 middleware |

**决策 S7-07**：**禁止** S0 引入 K8s 多集群（`10` §3.4、`过渡架构` §7）；S2 Cell **优先** 多 ECS + 多 RDS（与 `10` 升配路径一致），K8s 为 **S3 可选**。

## 6. Connector 两层实例模型（百 ERP / 百 OA）

```text
Layer A  ConnectorDefinition（全局/GIP）
         Blueprint YAML · CMV op · Silver/Gold 认证
         路径：integration/catalog/{system}/{vendor}/

Layer B  ConnectorInstance（per tenant）
         tenant_id + pack_id + env(prod|staging) + secrets_ref
         base_url Override merge · Edge Agent 绑定
```

| 决策 ID | 决策 |
|---------|------|
| S7-08 | **100 ERP** = ≤100 **Instance** 行，通常 **≤15 Definition**（金蝶/用友/SAP…）× 多 tenant |
| S7-09 | 凭证 **仅** Instance 级加密存储（KMS/ vault）；Runtime **按 tenant 解密**，禁止全局共享 token（iPaaS 多租户共识） |
| S7-10 | OA/IM 统一 `system=oa`；钉钉/企微/泛微 = **不同 pack_id**，非新 LegacySystem |
| S7-11 | 私网 ERP **必须** Edge Agent；S1 百级 = **Edge 连接池**（每 tenant 1～N agent），不扩大 L0 |

**执行面扩展（S2）**：`connector_worker` 池 **仅** 执行 httpx 调用 + 回传 snapshot；**编排、Rule、Revert 仍在** `execution_service`（闭合 `过渡架构` §5 connector-host 语义）。

## 7. 消息、Outbox 与队列选型

| 阶段 | 机制 |
|------|------|
| Core 1.0 / S0 | 同进程 `DomainEvent` handler（`DomainEvent.schema.json`） |
| S1 | **Transactional Outbox** 表 → 单进程 Worker（对账、通知、Evidence 异步组装） |
| S2 | Outbox → **FactoryOS Queue**；**审定默认实现 = Redis Streams**（阿里云 Redis，`10` 已有组件） |
| S3 | 高吞吐 Cell 可选 **NATS JetStream**；**仍禁止 Kafka 集群**（ADR-000 T-09、`10` §3.4） |

**决策 S7-12**：队列 **只** 用于 **异步副作用**（对账 Job、IM 通知、Analytics 管道）；**L2 写路径保持同步 + ExecutionRecord**，避免分布式事务侵蚀 ADR-002。

**决策 S7-19（审定 2026-06-16）**：S1+ 异步队列 **锁定 Redis Streams**；实现须通过 `FactoryOS Queue` 抽象，禁止业务代码直耦 Redis API。

**决策 S7-13**：跨 Cell **Analytics** 走 **只读事件汇**（Outbox → 全局 OLAP），Cell 间 **禁止** 互查 RDS（Cell 架构共识）。

## 8. 配额、限流与 License 扩展

在 ADR-003 `TenantLicense` 之上增加 **TenantQuota**（同 `license_service` 模块，S1 启用）：

| 配额键 | 说明 | S1 默认 |
|--------|------|---------|
| `max_factories` | License `factory_count` 硬 enforce | 按 SKU |
| `writes_per_day` | L2 写次数 | 10⁵/tenant/day |
| `connector_calls_per_min` | 按 tenant+pack 令牌桶 | 60/min |
| `mcp_tools_per_day` | MCP tools/call | 10⁴ |
| `llm_tokens_per_month` | 对齐 `12` 成本框架 | 按 SKU |
| `edge_agents_max` | 注册 Edge 数 | 5 |

**决策 S7-14**：超限 → **429** + Audit `quota.exceeded`；**不** silent drop。

**决策 S7-15**：`apps/api` middleware **顺序**：Auth → **tenant_id** → **Quota** → Rule → Execution。

## 9. 多租户隔离强化路径

| 阶段 | 机制 |
|------|------|
| S0 Gate 0 | 应用层 `WHERE tenant_id`（`多租户` MT-01） |
| S1（≥20 tenant 或百级启动） | **PostgreSQL RLS** 全表启用（闭合 MT-01 Y2 optional → S1 必做） |
| S1 Bridge | schema-per-tenant |
| Silo | 物理隔离 + 可选 per-tenant KMS CMK |

**决策 S7-16**：RLS 与 B-Lite `builtin_ledger` **不冲突** — 同策略，Path C 表集合不同（闭合 ADR-007 C6）。

**决策 S7-17**：MCP `/mcp/v1/{tenant_id}` URL tenant **必须** 与 JWT/API Key claim 一致（已有 `多租户` §3.3）；S2 按 Cell 发 **tenant-scoped** MCP endpoint DNS。

## 10. 可观测性与全链路追踪（规模态）

**不新增 L0「DataFlowService」**；S1 规模态 **强制**：

| 层 | 要求 |
|----|------|
| Logs | structlog **必含** `tenant_id`, `cell_id`, `trace_id`, `exec_id` |
| Traces | OpenTelemetry **必 export**（S1 起重 stub 废止）；span 必带 tenant 属性 |
| Audit | append-only；S2 **按 tenant 分区** 或冷归档 OSS |
| Evidence | `GET /evidence` 为合规主入口（E-09） |

**决策 S7-18**：Connector 实例级 **delivery 隔离** — A 厂 ERP 超时 **不** 占用 B 厂 worker 槽（独立 per-tenant 队列 depth 配置）。

---

## 11. L0 内核不变边界（铁律）

以下内容 **不** 随百级/千级变化：

```text
graph_service · rule_engine · execution_service · audit_service
写路径 · CMV ABI · Graph freeze · Rule deny-default · Revert 语义
```

**可演进（GIP / 基础设施）**：

```text
integration/* · connector_instances · Edge Agent · Studio
Cell Router · Outbox Worker · connector_worker 池 · Quota
```

**不可入 L0（闭合前序六问）**：

- 岗位习惯 / 个人日程 / 终端「变聪明」→ **L2 Skill Pack + Data-L2**（Evolution 宪章）
- 完整 OA BPM 审批引擎 → **conn-oa Pack + CMV**
- Process Mining 替代 ERP → **禁止**（Evolution EVO-X07）

## 12. 演进触发器与 Gate

| 触发条件 | 动作 | ADR 修订 |
|----------|------|----------|
| ≥20 tenant **或** RDS CPU >60% 14d | 启用 **RLS** + 读副本 | — |
| ≥50 tenant | 评估 **第 2 Cell**；Outbox Worker 独立进程 | — |
| ≥10² tenant | **Global Router** 上线；Quota 硬 enforce | — |
| connector p99 >3s 且 CPU 偏 connector | 抽 **connector_worker** 池 | 修订膨胀期 §3 |
| 单 tenant Legacy VPN | **Silo** 或 Bridge + Edge | — |
| ≥10³ tenant | Cell 数 ≥5；Analytics 事件汇 | 可选 S3 ADR |

**Gate 0 扩展**：`core-v1.0.0` = AC-BASE-001 **52 P0**（原 48 + **S-01～S-04** 规模预埋，ADR-007 §15）；S1 百级回归 **S1-SCALE-*** 另文档化。

**决策 S7-21（审定 2026-06-16）**：§15 五张表/列 **纳入 Gate 0 Alembic migration**（可空表、无业务数据）。

## 14. 后果

### 正面

- **一步到位预设**：W1 预埋字段即可通 S0，S1/S2 **无 L0 破坏性迁移**
- 百级 **Pool+RLS+Instance** 与千级 **Cell+Router** 同一条演进线，对齐 AWS/webMethods/iPaaS 主流
- 「100 ERP」工程上 = Instance 复制，**不改内核**
- Enterprise Silo 与 SMB Pool **同一 Package 导入**

### 负面 / 代价

- S1 起 **运维复杂度** 上升（RLS、Outbox、Cell 路由）
- Silo 客户 **成本线性**（与 webMethods spoke VPC 同理）
- 需 **S1-SCALE** 回归集与 Runbook 增补（S1 百级启用时）

## 15. W1 行动清单（Gate 0 migration，审定纳入）

1. `tenants` 表：`cell_id`, `placement_tier`, `region`（默认 `cell-default` / `pool`）
2. `connector_instances` + `tenant_quotas` + `outbox_events` 表（**空表 migration**）
3. Repository 层 **强制** tenant filter 接口（CI 已有 `check_import_boundaries.py`）
4. `TenantRegistry` 内存 stub：`tenant_id → cell-default`
5. `FactoryOS Queue` 接口；S0 **in-process**；S1 **Redis Streams** 实现（S7-19）
6. OpenAPI **不** 暴露 Cell 管理（控制面 **internal/admin** Y2）

验收：**AC-BASE-001 S-01～S-04**。
