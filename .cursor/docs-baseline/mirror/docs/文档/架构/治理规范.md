# 治理规范：模块包、DSL 与定制治理

> 版本：v0.1.0 | 日期：2026-06-16  
> 用途：防止 D2 定制腐蚀内核、防止 Pack 滥发、对齐商业与技术决策  
> 关联：[08-平台战略与两次交付模型](../../准备/2026-06-16/08-平台战略与两次交付模型.md)、[ADR-003](./架构决策记录-003-模块包授权与覆盖配置.md)

---

## 1. 治理目标

| 目标 | 手段 |
|------|------|
| 内核稳定 | Platform-L0 变更须 ADR |
| 定制可控 | Override + Pack；≥70% D2 入库 |
| 商业可追溯 | 新 SKU 先进 CAPABILITY-PACK-MAP |
| 灯塔不绑架 | D2 书面范围 + R2 ≤90 天 |

---

## 2. 变更分类与审批

| 类型 | 示例 | 审批 | 产出 |
|------|------|------|------|
| **G0 内核** | 新写 DSL、改 execution 流程 | 平台负责人 + ADR | ADR-00x |
| **G1 Pack** | 新 graph-* / cap-* / conn-* | 产品 + 研发 Lead | 模块包规格 登记 + MAP |
| **G2 Skill** | 新 skill-* FSM | 产品 + 实施代表 | AgentModule Schema |
| **G3 Override** | 产线阈值、字段 map | 实施 + 工厂 IT 签字 | OverrideDocument |
| **G4 L3 项目** | 无法 Pack/Override 的一次性 | 商务 + 平台负责人 | 合同附录；≤30% D2 |

**默认路径**：G3 → G2 → G1；**禁止** 直接 G4 改内核。

---

## 3. D2 定制决策树

```text
客户特异需求
  ├─ 能否用现有 Pack + Override？ → 是 → G3，结案
  ├─ 能否新 Skill/Graph 版本？   → 是 → G2/G1，≥70% 目标
  ├─ 能否新 Connector 映射？     → 是 → conn Pack 升级
  └─ 以上皆否                   → G4 L3 项目（记录原因，季度复盘是否可 Pack 化）
```

---

## 4. Pack 入库评审（08 §6.3 + 模块包规格 M3+）

**入库条件**（全部满足）：

1. 生产环境 ≥30 天
2. 验收用例 ≥5 条
3. 不违反 ADR-002
4. 可配置启用，无 `tenant_id === 'xxx'` 硬编码
5. CAPABILITY-PACK-MAP 已更新

**评审人**：研发 Lead + 实施 Lead；每月 Pack 例会。

---

## 5. DSL 新增流程

1. 提案：业务场景 + 安全级别（L0–L3）
2. 检查：ADR-002 首期禁止列表
3. 设计：`DSLAction` Schema + Compensator（L2 写）
4. 绑定：`cap-*` Pack
5. 测试：contract test + revert 单测
6. 文档：DSL规格 更新

---

## 6. 灯塔客户（R5）特殊规则

- 内部子公司 **按外部客户** 走 D1/D2 Gate
- 「兄弟公司顺便加」→ 默认 **D2 新范围**，不插队 Phase 1
- 无 D1 签字不算「试点成功」
