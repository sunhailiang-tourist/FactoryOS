## 1. 背景

FactoryOS 在 Coding 前已形成 **46 份** Markdown 规格与战略文档（含 prepare 9 份、docs 工程契约、Blueprint、Connector 预留目录）。本文档是对全库的 **架构决策审阅汇总**，将分散结论收敛为可审定、可执行、可审计的 **一条决策链**。

**审阅结论**：文档层已满足 Gate 0（底座可 Coding）与三年商业—技术并行 Blueprint；**执行层**（repo 骨架、AC-BASE-001 测试、mock Connector）尚待 Coding 落地。

## 2. 产品与市场决策

### 2.1 我们是什么 / 不是什么

| 决策 ID | 决策 | 来源 |
|---------|------|------|
| P-01 | FactoryOS 是制造业 **AI 执行平台 OS**（Overlay），**不是**聊天 Agent 产品本体 | 00 §1, 08 §1 |
| P-02 | **不替换、不改造** 客户 ERP/MES/WMS/OA 账本；只做跨系统 governed 执行与编排 | 00 §1, 06 §二 |
| P-03 | Agent 仅为 **交互入口**；产品壁垒在 frozen Graph + Rule + DSL + Audit + Revert | 00 §1, 08 §2 |
| P-04 | 对外主品牌 **FactoryOS**；`Factory-AI-Native-System` 仅内部架构名 | 00 §1 |
| P-05 | 当前阶段 **不先做对外营销**；以哈森体系灯塔部署验证可靠性 | 08 §1 |
| P-06 | **产品 B**（全栈 AI Factory OS）、**C 类客户**（无系统）**暂缓** | 00 §2, 01 |

### 2.2 客户与场景优先级

| 决策 ID | 决策 | 来源 |
|---------|------|------|
| P-07 | 客户分型：**A**（有 ERP/MES，优先）> **B**（烂系统+Excel）> **C**（暂缓） | 01 |
| P-08 | 场景顺序：**报工 → 生产分析 → 老板决策**；均先只读/建议/确认 | 00 §3 |
| P-09 | 首期 **禁止**：自动财务记账、付款、成本结转、无确认库存调整、无确认自动审批 | 00 §3, ADR-002 |
| P-10 | 操作铁律：**先读 → 后建议 → 再确认 → 最后自动化** | 00 §3 |

### 2.3 验证与商业路径

| 决策 ID | 决策 | 来源 |

## 13. 后果

### 正面

- 全库决策有 **单一审定入口**（本文档 + ADR-001～006）
- Coding、商务、实施可引用同一决策 ID（如 D-05、C-03、M-01）
- 三年商业与技术并行有正式追溯，半年内方向性返工风险 **低**

### 负面 / 代价

- 文档体量大；**变更须走 GOVERNANCE**，不可口头漂移
- Phase 1 需维护 Pack/License stub，略增早期复杂度
- 真实业务与演示能力 **依赖 Gate 0' 工厂输入**，不能指望纯底座文档产出可卖 Demo
